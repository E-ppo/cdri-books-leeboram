# 개선사항

<summary><strong>상세 검색 후 검색창 입력어 처리</strong></summary>

### 초기 구현

최초에는 상세 검색을 통해 검색을 실행하면, 상단 메인 검색창(SearchBar)의 입력값을 비우도록 구현했습니다.

```ts
const handleFullSearch = (params: SearchParams) => {
  setKeyword(params.keyword);
  setDetailSearchKey(prev => prev + 1);
  addHistory({ keyword: params.keyword, category: params.category });
  onSearch(params.keyword, CATEGORY_TO_TARGET[params.category] ?? DEFAULT_TARGET);
};

const handleDetailSearch = (params: SearchParams) => {
  setKeyword(''); // 메인 검색창 초기화
  addHistory({ keyword: params.keyword, category: params.category });
  onSearch(params.keyword, CATEGORY_TO_TARGET[params.category] ?? DEFAULT_TARGET);
};
```

### 초기 의도

과제 요구사항 중 "전체 검색과 상세 검색은 동시에 진행 불가"라는 조건을 만족시키기 위해, 상세 검색이 수행된 순간에는 메인 검색창 입력값을 비워 두 검색 상태가 시각적으로도 충돌하지 않도록 처리했습니다.

### UX 관점에서의 재검토

그러나 일반적인 검색 UX 표준에 비추어 보면, 검색 수행 후 검색창에 입력어를 유지하는 것이 더 바람직합니다.

**검색어를 유지해야 하는 이유:**

1. **컨텍스트 유지** — 사용자가 방금 무엇을 검색했는지 결과 화면에서 즉시 확인할 수 있습니다.
2. **수정 용이성** — 오타를 수정하거나 키워드의 일부만 변경해 재검색하기 쉽습니다. (예: "리액트" → "리액트 훅")
3. **결과와 입력의 연결성** — 검색 결과와 입력어의 관계가 명확해져 사용자의 인지 부하가 줄어듭니다.

**검색어가 사라지는 것이 적절한 경우:**

- 채팅 메시지 입력창처럼 "전송 후 초기화"가 맥락에 맞는 경우
- 모바일에서 재검색 빈도가 매우 낮은 특수한 플로우

구글, 네이버, 아마존, 유튜브 등 대부분의 검색 UI가 입력어를 유지하는 방식을 채택하고 있습니다. 도서 검색 페이지의 성격을 고려하면 입력어를 유지하는 방향이 보다 표준적인 UX에 부합합니다.

### 공식적인 UX 근거

이 판단은 개인적인 취향이 아니라 공식적인 UX 원칙과 연구에 기반합니다.

#### 1. Jakob Nielsen의 10가지 사용성 휴리스틱 (Nielsen Norman Group)

- **#1 Visibility of system status (시스템 상태의 가시성)**
  시스템은 현재 어떤 상태인지 사용자에게 적절한 피드백을 통해 항상 알려야 합니다. 검색 결과 화면에서 "현재 어떤 검색어에 대한 결과를 보고 있는지"는 시스템 상태에 해당하며, 검색창에 입력어가 남아 있는 것이 이 원칙을 만족시킵니다.

- **#6 Recognition rather than recall (기억보다 인식)**
  사용자가 정보를 기억해내도록 강요하지 말고, 화면에 보여줘야 합니다. 검색창을 비우면 사용자는 방금 자신이 입력한 검색어를 다시 떠올려야 합니다.

참고: https://www.nngroup.com/articles/ten-usability-heuristics/

#### 2. Baymard Institute의 커머스 검색 UX 연구

Baymard Institute는 대규모 사용성 테스트를 기반으로 커머스 검색 경험에 대한 가이드라인을 발표해 왔으며, 검색 결과 페이지(SERP)에서 "사용자가 입력한 검색 쿼리는 검색창에 그대로 유지되어야 한다"는 점을 반복적으로 강조합니다.

참고: https://baymard.com/ (Search & Navigation 관련 연구 다수)

#### 3. 플랫폼 디자인 시스템

- **Material Design**: Search 패턴에서 검색 수행 후에도 쿼리를 입력 필드에 유지할 것을 권장
- **Apple Human Interface Guidelines (HIG)**: Search fields에서 최근 쿼리의 가시성을 유지하는 방향을 표준 동작으로 설명

### 요구사항 재해석

요구사항 "전체 검색과 상세 검색은 동시에 진행 불가"는 **검색 로직의 배타성**(한 번에 하나의 검색 결과만 활성)을 의미하는 것으로 재해석할 수 있습니다. 즉, **상세 검색 실행 시 기존 검색 결과가 새 결과로 교체되는 것만으로도 이 조건은 충족되며**, 검색어를 메인 입력창에 표시하는 것은 요구사항과 충돌하지 않습니다.

### 개선 적용

위의 UX 원칙(Nielsen 휴리스틱 #1, #6)과 요구사항 재해석을 근거로, 상세 검색 이후에도 검색어가 메인 검색창에 그대로 남도록 수정했습니다.

또한 동작이 동일해진 `handleFullSearch`와 `handleDetailSearch`를 하나의 `handleSearch`로 통합했습니다.

관련 파일: `src/pages/Search/components/SearchHeader/SearchHeader.tsx`

```ts
const handleSearch = (params: SearchParams) => {
  setKeyword(params.keyword);
  setDetailSearchKey(prev => prev + 1);
  addHistory({ keyword: params.keyword, category: params.category });
  onSearch(params.keyword, CATEGORY_TO_TARGET[params.category] ?? DEFAULT_TARGET);
};
```

**개선 효과**

- 사용자가 현재 어떤 검색어의 결과를 보고 있는지 즉시 인지 가능
- 검색어 일부만 수정해 재검색하기 쉬워짐
- 중복 로직 제거로 코드 가독성 및 유지보수성 향상
- **요구사항 충족 확인: 상세 검색 실행 시 기존 검색 결과가 즉시 새 결과로 교체되므로, 두 검색이 동시에 활성화되지 않음 (배타성 유지)**

<summary><strong>SearchBar 접근성 및 사용자 경험 강화</strong></summary>

기본 요구사항인 "검색 기능 구현"을 넘어, 접근성(A11y)과 인터랙션 품질 측면에서 다음과 같은 기능을 **추가**로 구현했습니다.

### 접근성(A11y) — 스크린리더·키보드 사용자 지원

| 항목 | 구현 내용 |
|---|---|
| **ARIA Combobox 패턴** | input에 `role="combobox"`, `aria-autocomplete="list"`, `aria-expanded`, `aria-controls`, `aria-activedescendant` 속성 적용 |
| **ARIA Listbox 패턴** | `<ul role="listbox">` + `<li role="option" aria-selected>` 구조, `useId()` 기반으로 안정적인 id 연결 |
| **접근 가능한 이름(Accessible Name)** | input에 `aria-label="도서 검색"`, 삭제 버튼에 `aria-label="{키워드} 검색 기록 삭제"` 지정 |
| **검색 랜드마크** | 루트 컨테이너에 `role="search"` 부여 — 스크린리더 사용자가 랜드마크 단위로 바로 이동 가능 |
| **장식 아이콘 숨김** | 검색·닫기 아이콘에 `aria-hidden="true"` — 보조기기 중복 읽기 방지 |

### UX — 인터랙션 품질

| 항목 | 구현 내용 |
|---|---|
| **키보드 네비게이션** | ↑/↓ 키로 검색 기록 탐색, 끝에서 처음으로 순환(wrap-around), Enter로 선택·제출 |
| **한글 IME 안전 처리** | `isComposing` 상태와 `onCompositionStart/End` 훅으로 한글 조합 중 Enter를 무시 → 마지막 글자 누락 방지 |
| **스마트 blur 처리** | `relatedTarget`이 컨테이너 내부면 히스토리 유지 → Tab 이동이나 삭제 버튼 클릭 시 히스토리가 닫히지 않음 |
| **제출 후 자동 blur** | Enter 제출 시 `inputRef.blur()` 호출 → 모바일 소프트 키보드가 자동으로 내려가 결과 확인이 즉시 가능 |
| **선택/삭제 액션 분리** | 검색 기록 항목에 선택 버튼과 삭제 버튼을 별도 `<button>`으로 분리 — 각각 독립적으로 포커스·조작 가능 |
| **활성 항목 시각 표시** | `activeIndex` 기반 배경 하이라이트 — 키보드 사용자가 현재 위치를 명확히 인지 |

### 적용된 표준·가이드라인 근거

- **WCAG 2.1.1 Keyboard** — 모든 기능을 키보드만으로 조작 가능
- **WCAG 2.4.3 Focus Order** — 포커스 이동 중 UI가 예기치 않게 닫히지 않음
- **WCAG 4.1.2 Name, Role, Value** — 모든 인터랙티브 요소에 이름과 역할 제공
- **WAI-ARIA Authoring Practices** — Combobox with Listbox Popup 패턴 준수
- **인클루시브 디자인(Inclusive Design)** — 한글 IME 사용자 대상 선제적 이슈 대응

<summary><strong>전체 컴포넌트 접근성 일관화</strong></summary>

SearchBar 강화 후 "접근성은 일관성이 핵심"이라는 원칙에 따라, 나머지 커스텀 컴포넌트에도 동일한 기준을 적용했습니다. 각 컴포넌트의 성격에 맞는 WAI-ARIA 패턴을 선택해 최소 침습적으로 보강했습니다.

### 1. DetailSearchPopup — Dialog 패턴

검색 결과에 영향을 주는 중요한 상호작용이므로 완전한 모달 다이얼로그 패턴을 적용했습니다.

| 항목 | 구현 내용 |
|---|---|
| **Dialog 시맨틱** | `role="dialog"`, `aria-modal="true"`, `aria-labelledby={titleId}` |
| **시각적으로 숨긴 제목** | `<h2 class="sr-only">상세 검색</h2>` — 스크린리더가 다이얼로그 이름 안내 |
| **포커스 진입** | 팝업이 열리면 `inputRef.current?.focus()`로 검색 입력에 자동 포커스 |
| **포커스 복귀** | 닫힐 때 `previousFocusRef`에 저장된 트리거로 포커스 반환 (WCAG 2.4.3) |
| **포커스 트랩** | `Tab`/`Shift+Tab`이 팝업 경계를 넘어가지 않도록 첫/마지막 포커스 요소 순환 처리 |
| **Escape로 닫기** | 다이얼로그 내 `keydown` 리스너에서 `Escape` 감지 후 닫기 |
| **트리거 상태 노출** | 트리거 버튼에 `aria-expanded`, `aria-haspopup="dialog"`, `aria-label` 부여 |
| **닫기 버튼** | `aria-label="상세 검색 닫기"` |

파일: `src/components/DetailSearchPopup/DetailSearchPopup.tsx`

### 2. Icon — 기본 장식 정책

전체 앱에서 Icon 컴포넌트가 광범위하게 사용되므로, **기본값을 장식(decorative)으로 설정**해 모든 사용처에서 일관되게 스크린리더 잡음이 발생하지 않도록 했습니다.

- **기본 동작:** `aria-hidden="true"` 자동 적용 (span과 SVG 양쪽)
- **의미 있는 아이콘:** 소비자가 `aria-label="..."`을 전달하면 `aria-hidden`을 해제하고 `role="img"` 자동 부여
- **명시적 오버라이드 허용:** `aria-hidden` prop을 명시적으로 넘기면 우선 적용

이 정책 덕분에 Dropdown의 `ChevronDown`, Header의 활성 탭 인디케이터 등 모든 장식 아이콘이 자동으로 보조기기에서 무시됩니다.

파일: `src/components/Icon/Icon.tsx`

### 3. Accordion — Disclosure 패턴

WAI-ARIA Authoring Practices의 Disclosure 패턴을 적용해 트리거와 콘텐츠 영역을 명확히 연결했습니다.

| 항목 | 구현 내용 |
|---|---|
| **ID 자동 생성** | `Item` 내부에서 `useId()`로 `triggerId`/`contentId` 생성 후 Context로 전달 |
| **Trigger → Content 연결** | Trigger에 `id`, `aria-expanded`, `aria-controls={contentId}` |
| **Content 영역 의미 부여** | Content에 `id`, `role="region"`, `aria-labelledby={triggerId}` |
| **비-버튼 경로 키 지원** | `as="div"`로 커스텀 렌더 시 `Enter`/`Space`에서 토글 동작 (네이티브 `<button>`은 자동) |

파일: `src/components/Accordion/Accordion.tsx`, `Accordion.types.ts`

### 4. StatusMessage — Live Region

로딩/결과 없음/에러 등 동적으로 나타나는 상태 메시지이므로 스크린리더가 감지·안내할 수 있도록 live region으로 구성했습니다.

- 루트에 `role="status"`, `aria-live="polite"` 부여
- 장식용 아이콘 래퍼에 `aria-hidden="true"`

파일: `src/components/StatusMessage/StatusMessage.tsx`

### 5. Header — 장식 요소 숨김

- 활성 탭 위치를 따라 움직이는 바 인디케이터(`<span>`)에 `aria-hidden="true"` 추가
- NavLink는 React Router가 활성 링크에 `aria-current="page"`를 자동 부여하므로 별도 처리 불필요

파일: `src/components/Header/Header.tsx`

### 적용된 표준·가이드라인 근거

- **WAI-ARIA Authoring Practices**
