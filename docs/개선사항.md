# 개선사항

요구사항을 충족하면서 동시에 사용자 경험·접근성·유지보수성 측면에서 추가로 고려한 결정들을 정리한 문서입니다. 각 섹션은 **배경 → 접근 → 구현 포인트 → 효과** 순으로 구성되어 있습니다.

<details>
<summary><strong>상세 검색 후 검색창 입력어 처리</strong></summary>

### 배경

요구사항에 "전체 검색과 상세 검색은 동시에 진행 불가"라는 조건이 있습니다. 이 조건을 만족시키기 위해 최초 구현에서는 상세 검색 실행 시 메인 검색창의 입력값을 비웠습니다. 그러나 일반적인 검색 UX 표준에 비추어 보면, 검색 수행 후에도 입력어를 검색창에 유지하는 것이 더 바람직합니다.

### 접근

요구사항을 **검색 로직의 배타성**(한 번에 하나의 검색 결과만 활성)으로 재해석했습니다. 상세 검색 실행 시 기존 결과가 새 결과로 교체되는 것만으로도 배타성 조건은 충족되며, 검색어를 메인 입력창에 표시하는 것은 요구사항과 충돌하지 않습니다. 이 근거로 상세 검색 후에도 검색어가 메인 검색창에 그대로 남도록 수정했고, 동작이 동일해진 두 핸들러를 하나로 통합했습니다.

### 구현 포인트

```ts
const handleSearch = (params: SearchParams) => {
  setKeyword(params.keyword);
  setDetailSearchKey(prev => prev + 1);
  addHistory({ keyword: params.keyword, category: params.category });
  onSearch(params.keyword, CATEGORY_TO_TARGET[params.category] ?? DEFAULT_TARGET);
};
```

- 상세 검색 후 메인 검색창 입력어 초기화 로직 제거
- `handleFullSearch`·`handleDetailSearch` → `handleSearch` 하나로 통합
- 요구사항 "동시 진행 불가"는 결과 교체(배타성)로 충족

### 효과

- 사용자가 현재 어떤 검색어의 결과를 보고 있는지 즉시 인지 가능
- 검색어 일부만 수정해 재검색하기 쉬워짐
- 중복 로직 제거로 코드 가독성 및 유지보수성 향상
- **요구사항 충족 확인:** 상세 검색 실행 시 기존 검색 결과가 즉시 새 결과로 교체되므로, 두 검색이 동시에 활성화되지 않음 (배타성 유지)

### 근거

- **Jakob Nielsen의 10가지 사용성 휴리스틱**
  - #1 Visibility of system status — 현재 어떤 상태인지 항상 사용자에게 알린다
  - #6 Recognition rather than recall — 기억하게 하지 말고 화면에 보여준다
  - 참고: https://www.nngroup.com/articles/ten-usability-heuristics/
- **Baymard Institute** — 검색 결과 페이지에서 입력 쿼리를 검색창에 유지하도록 권장 (https://baymard.com/)
- **Material Design / Apple HIG** — 검색 수행 후 쿼리 유지를 표준 동작으로 제시
- 실사용 사례 — 구글, 네이버, 아마존, 유튜브 등 주요 서비스가 입력어를 유지

<summary><strong>검색 결과 페이지네이션 — 무한 스크롤로 구현</strong></summary>

### 배경

요구사항은 "검색어 입력 후 Enter키를 누르면 검색이 진행되고 10개의 결과가 화면에 노출 (페이지 당 10개 아이템)"입니다. 데이터 요청 단위는 10개로 유지하되, 사용자 경험 측면에서는 페이지 이동보다 연속적인 탐색이 자연스럽다고 판단했습니다.

### 접근

**10개 단위로 API를 호출한다는 계약은 유지**하되, UI는 **페이지 번호 대신 무한 스크롤**로 구현했습니다. 데이터 요청 단위는 요구사항 그대로이고, UI는 페이지 전환의 맥락 단절 없이 결과를 탐색할 수 있도록 개선했습니다.

### 구현 포인트

| 항목            | 구현 내용                                                                                                                       |
| --------------- | ------------------------------------------------------------------------------------------------------------------------------- |
| **데이터 계층** | `useInfiniteQuery`(TanStack Query)로 페이지 단위 캐시 관리, `getNextPageParam`에서 `meta.is_end` 기준으로 다음 페이지 여부 판단 |
| **요청 단위**   | **Kakao Book Search API의 기본 `size=10`을 사용해 페이지당 10개 단위로 호출** (요구사항 준수)                                   |
| **트리거**      | `IntersectionObserver` 기반 `useInfiniteScroll` 훅 — 목록 끝 sentinel이 뷰포트 진입 시 `fetchNextPage` 호출                     |
| **로딩 상태**   | `isFetchingNextPage` 플래그로 추가 페이지 로딩을 시각적으로 구분                                                                |

### 효과

- **탐색 흐름의 연속성** — 페이지 버튼을 누르는 컨텍스트 스위칭 없이 스크롤만으로 결과 확인
- **모바일 친화적** — 터치 환경에서 페이지 번호 선택보다 자연스러움
- **빠른 첫 응답** — 10개 단위 호출로 첫 화면 표시가 빠름
- **이탈 지점 감소** — 페이지 이동 시 생기는 로딩 공백이 없어 결과 확인 흐름이 끊기지 않음
- **요구사항 충족 확인:** "페이지당 10개 아이템" 계약은 API 호출 단위로서 그대로 준수

<summary><strong>반응형 디자인 — 모바일 · 태블릿 · 데스크탑 대응</strong></summary>

### 배경

도서 검색은 스마트폰·태블릿·데스크탑 어디서든 자주 사용되는 패턴입니다. 단일 디자인을 모든 화면 폭에 억지로 맞추기보다, 세 단계 브레이크포인트에 맞춰 레이아웃·타이포그래피·상호작용을 조정해 각 기기에서 자연스럽게 사용할 수 있도록 설계했습니다.

### 접근

Tailwind 기본 브레이크포인트(`sm`/`md`/`lg`)를 채택해 별도 설정 없이 일관되게 사용합니다. 콘텐츠·터치 타겟·한글 타이포그래피를 우선 고려하고, 가능한 한 CSS 브레이크포인트로 해결하며 JS 레벨 대응(`useMediaQuery`)은 동작 분기가 꼭 필요한 경우에만 사용합니다.

### 구현 포인트

| 단계             | 기준                  | 레이아웃 특징                                               |
| ---------------- | --------------------- | ----------------------------------------------------------- |
| **기본(모바일)** | `< 640px`             | 단일 열, 상세 검색은 필터 아이콘으로 축약, 이미지·폰트 축소 |
| **sm / md**      | `≥ 640px` / `≥ 768px` | 확장 레이아웃, "상세 검색" 버튼 노출, 이미지 중간 크기      |
| **lg**           | `≥ 1024px`            | 헤더 가로 정렬, 카드 가로 배치, 이미지·여백 최대            |

**대표 사례**

- 상세 검색 트리거 전환: 모바일은 필터 아이콘, 태블릿부터는 "상세 검색" 버튼 노출
  ```tsx
  <button className="sm:hidden ..."><Icon name="Filter" size={22} /></button>
  <Button className="hidden sm:flex" size="sm" variant="outline">상세 검색</Button>
  ```
- 도서 카드: 모바일 세로 스택 → 태블릿·데스크탑 가로 배치, 이미지 크기 3단계
  ```tsx
  <div className="flex flex-col md:flex-row gap-4">
    <img className="w-24 md:w-36 lg:w-52.5 ..." />
  </div>
  ```
- 헤더: 데스크탑에서 가로 정렬 + 절대 위치 중앙 배치, 모바일은 세로 스택
- 타이포그래피: `body1-bold sm:title2` 등 폭에 따라 단계적 확대
- JS 레벨 분기: `useMediaQuery('(min-width: 640px)')` 결과로 팝오버 배치 방향 전환
  - `matchMedia` 직접 구독 (의존성 없음)
  - 초기값 동기 계산으로 hydration mismatch 방지
  - 이벤트 리스너 cleanup을 단위 테스트로 검증

### 효과

- **기기별 자연스러운 UI** — 각 화면에서 우선순위 콘텐츠가 먼저 드러남
- **터치 타겟 유지** — 모바일에서도 버튼·링크 클릭 영역 확보
- **한글 줄바꿈 품질** — `break-keep-all`로 단어 단위 줄바꿈 보장
- **JS 의존 최소화** — 대부분 CSS로 해결해 초기 렌더 성능·SSR 호환성 확보

<summary><strong>Compound Component 패턴으로 Accordion 설계</strong></summary>

### 배경

도서 상세 정보(책 소개, 목차 등)에 접히고 펼쳐지는 UI가 필요했습니다. 단일 Props 방식(`<Accordion title={...} content={...} />`)으로도 요구사항은 충족되지만, 마크업 자유도·다중 아이템 확장·다형성 지원이 어려워 유연성이 떨어집니다.

### 접근

**Compound Component 패턴**으로 설계해 React Context로 상태를 공유하고 서브 컴포넌트(`Item`/`Trigger`/`Content`)가 각자의 역할만 담당하도록 분리했습니다. 소비자는 조합만으로 원하는 구조를 만들 수 있습니다.

```tsx
<Accordion multiple>
  <Accordion.Item value="intro">
    <Accordion.Trigger>책 소개</Accordion.Trigger>
    <Accordion.Content>{book.contents}</Accordion.Content>
  </Accordion.Item>
  <Accordion.Item value="toc">
    <Accordion.Trigger as="div">목차</Accordion.Trigger>
    <Accordion.Content>{book.toc}</Accordion.Content>
  </Accordion.Item>
</Accordion>
```

### 구현 포인트

| 항목                     | 구현 내용                                                                                                              |
| ------------------------ | ---------------------------------------------------------------------------------------------------------------------- |
| **Context 이중화**       | `AccordionContext`(전체 상태) + `AccordionItemContext`(아이템별 상태) 분리 — Trigger/Content가 자신이 속한 Item만 조작 |
| **상태 관리 유연성**     | `multiple` prop으로 단일/다중 펼침 전환, 내부는 `Set<string>`으로 관리                                                 |
| **함수형 children 지원** | `Item`의 children에 `(isOpen) => ReactNode` 허용 — 상태에 따라 다른 트리거 UI 렌더 가능                                |
| **다형성 Trigger**       | `as="button"`(기본) 또는 `as="div"` — `div`일 때 `role="button"`·`tabIndex={0}`·Enter/Space 핸들러 자동                |
| **접근성 자동 연결**     | `useId()`로 `triggerId`/`contentId` 생성 → `aria-controls`·`role="region"`·`aria-labelledby` 자동 적용                 |
| **Context 가드**         | `useAccordionContext` 등에서 미적용 시 명시적 에러 throw — 오용을 런타임에 즉시 감지                                   |

### 효과

- **마크업 자유도** — 소비자가 원하는 아이콘·레이아웃·스타일 자유 조합
- **확장성** — 아이템 수가 늘어도 props가 증가하지 않음
- **일관된 접근성** — 모든 인스턴스가 ARIA·키보드 지원을 자동 획득
- **테스트 용이성** — 각 서브 컴포넌트의 단일 책임으로 단위 테스트 단순화

<summary><strong>Persist 스토어로 사용자 경험 연속성 확보</strong></summary>

### 배경

검색 기록과 찜한 책 목록은 사용자가 **세션을 넘나들며 기대하는 상태**입니다. 브라우저를 새로고침하거나 다음날 다시 접속했을 때 방금 찜해둔 책이 사라져 있다면 앱을 신뢰하기 어렵습니다. 상태를 메모리에만 두지 않고 영속화할 필요가 있었습니다.

### 접근

Zustand `persist` 미들웨어로 localStorage에 자동 동기화했습니다. 검색 기록과 위시리스트는 **관심사를 분리해 스토어 2개로 분할**했고, 각 스토어는 도메인 규칙(중복 방지, 최대 개수, 토글)을 자체적으로 캡슐화합니다.

| 항목                 | 결정                                  | 이유                                                     |
| -------------------- | ------------------------------------- | -------------------------------------------------------- |
| **저장소**           | `localStorage`                        | 세션 간 영속성 필요, 민감 정보 아님                      |
| **스토어 분할**      | `searchHistoryStore`, `wishlistStore` | 검색 기록 갱신이 위시리스트 구독자를 리렌더링하지 않도록 |
| **스토리지 키 분리** | `search-history`, `wishlist`          | 도메인 독립 관리·디버깅·마이그레이션                     |

### 구현 포인트

**검색 기록** — 중복 방지, 최신 우선, 최대 8개 유지

```ts
const MAX_HISTORY_COUNT = 8;

addHistory: item =>
  set(state => {
    const isDuplicate = state.history.some(
      h => h.keyword === item.keyword && h.category === item.category
    );
    if (isDuplicate) return state;

    const newItem = { ...item, id: crypto.randomUUID(), timestamp: Date.now() };
    const updated = [newItem, ...state.history].slice(0, MAX_HISTORY_COUNT);
    return { history: updated };
  }),
```

**위시리스트** — ISBN 기준 동등성, 순수 토글 API

```ts
toggleWishlist: book =>
  set(state => {
    const exists = state.books.some(b => b.isbn === book.isbn);
    return {
      books: exists
        ? state.books.filter(b => b.isbn !== book.isbn)
        : [...state.books, book],
    };
  }),

isWishlisted: isbn => get().books.some(b => b.isbn === isbn),
```

- 각 항목에 UUID·타임스탬프 부여 → 삭제·정렬 안전
- `isWishlisted(isbn)` 셀렉터로 각 카드가 필요한 정보만 구독
- JSON 직렬화 가능한 자료구조만 사용(Set/Map 미사용) → persist 호환성

### 효과

- **새로고침 저항성** — 실수로 새로고침해도 방금 찜한 책이 남아 있음
- **세션 연속성** — 다음날 다시 접속해도 관심 도서 목록 유지
- **재검색 편의** — 이전 입력 검색어를 기억해 한 번의 클릭으로 재사용
- **도메인 규칙 단위 테스트 보장** — `searchHistoryStore.test.ts`, `wishlistStore.test.ts`로 중복 방지·최대 개수·토글·삭제 검증

<summary><strong>SearchBar 접근성 및 사용자 경험 강화</strong></summary>

### 배경

검색은 이 앱의 핵심 기능입니다. 키보드만 사용하는 사용자, 스크린리더 사용자, 한글 IME 사용자 등 다양한 사용 맥락에서 문제없이 동작하도록 접근성·인터랙션 품질을 요구사항 이상으로 강화했습니다.

### 접근

- **스크린리더/키보드 지원** — WAI-ARIA Authoring Practices의 Combobox with Listbox Popup 패턴을 적용
- **인터랙션 품질** — 한글 IME, 포커스 관리, 모바일 키보드 등 실사용 맥락에서의 세부 동작 정비

### 구현 포인트

**접근성(A11y)**

| 항목                   | 구현 내용                                                                                                        |
| ---------------------- | ---------------------------------------------------------------------------------------------------------------- |
| **ARIA Combobox 패턴** | input에 `role="combobox"`, `aria-autocomplete="list"`, `aria-expanded`, `aria-controls`, `aria-activedescendant` |

| **검색 랜드마크**      | 루트에 `role="search"` — 스크린리더 랜드마크 단위 이동 지원                                                      |

| 항목                    | 구현 내용                                                                             |
| ----------------------- | ------------------------------------------------------------------------------------- |

| **스마트 blur**         | `relatedTarget`이 컨테이너 내부면 히스토리 유지 — Tab/삭제 버튼 클릭 시 닫히지 않음   |

| **선택/삭제 분리**      | 항목에 선택 버튼과 삭제 버튼을 별도 `<button>`으로 분리                               |
| **활성 항목 시각 표시** | `activeIndex` 기반 배경 하이라이트                                                    |

<summary><strong>전체 컴포넌트 접근성 일관화</strong></summary>

### 배경

SearchBar에 접근성 강화 작업을 완료했지만, 접근성은 일관성이 핵심입니다. 다른 컴포넌트가 미흡하면 키보드·스크린리더 사용자는 여전히 장벽을 만납니다. 특히 DetailSearchPopup(모달), Icon(전체 아이콘 기반), Accordion(공개/비공개 패턴)은 WAI-ARIA APG 기준에서 필수 처리가 빠져 있었습니다.

### 접근

각 컴포넌트의 성격에 맞는 표준 ARIA 패턴을 선택해 **최소 침습적**으로 보강했습니다. 기존 동작·API는 유지하면서 속성·포커스 관리·키보드 핸들러만 추가하는 방향으로 진행했습니다.

### 구현 포인트

**1. DetailSearchPopup — Dialog 패턴**

| 항목                 | 구현 내용                                                         |
| -------------------- | ----------------------------------------------------------------- |
| **Dialog 시맨틱**    | `role="dialog"`, `aria-modal="true"`, `aria-labelledby={titleId}` |
| **시각적 숨김 제목** | `<h2 class="sr-only">상세 검색</h2>`                              |
| **포커스 진입**      | 팝업 열림 시 `inputRef.current?.focus()` 자동 포커스              |
| **포커스 복귀**      | 닫힘 시 `previousFocusRef`에 저장한 트리거로 복귀 (WCAG 2.4.3)    |
| **포커스 트랩**      | `Tab`/`Shift+Tab`이 경계를 넘지 않도록 첫/마지막 요소 순환        |
| **Escape로 닫기**    | 다이얼로그 내 `keydown` 리스너에서 Escape 감지                    |
| **트리거 상태 노출** | `aria-expanded`, `aria-haspopup="dialog"`, `aria-label`           |
| **닫기 버튼**        | `aria-label="상세 검색 닫기"`                                     |

**2. Icon — 기본 장식 정책**

- 기본 동작: `aria-hidden="true"` 자동 적용 (span + SVG 양쪽)
- 의미 있는 아이콘: `aria-label` 전달 시 `aria-hidden` 해제 + `role="img"` 자동
- 명시적 오버라이드 허용: `aria-hidden` prop을 직접 넘기면 우선 적용
- Dropdown의 `ChevronDown`, Header 활성 탭 인디케이터 등 모든 장식 아이콘이 자동으로 보조기기에서 무시됨

**3. Accordion — Disclosure 패턴**

| 항목                       | 구현 내용                                                           |
| -------------------------- | ------------------------------------------------------------------- |
| **ID 자동 생성**           | `Item` 내부 `useId()`로 `triggerId`/`contentId` 생성 → Context 전달 |
| **Trigger → Content 연결** | Trigger에 `id`, `aria-expanded`, `aria-controls={contentId}`        |
| **Content 의미 부여**      | Content에 `id`, `role="region"`, `aria-labelledby={triggerId}`      |
| **비-버튼 경로 키 지원**   | `as="div"` 시 Enter/Space에서 토글 (네이티브 button은 자동)         |

**4. StatusMessage — Live Region**

- 루트에 `role="status"`, `aria-live="polite"` — 동적 상태 메시지를 스크린리더에 전달
- 장식 아이콘 래퍼에 `aria-hidden="true"`

**5. Header — 장식 요소 숨김**

- 활성 탭 바 인디케이터(`<span>`)에 `aria-hidden="true"`
- NavLink는 React Router가 `aria-current="page"` 자동 부여
